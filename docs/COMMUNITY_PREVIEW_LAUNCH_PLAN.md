# Community Preview Launch Plan

Status: public feedback workflow for a Sepolia prototype. This plan does not approve a token offering, mainnet deployment, active guest benefits, or independent audit completion.

## Launch Objective

Turn the public documentation, website, and release-candidate materials into a usable feedback loop for reviewers, contributors, and community members.

The preview should help answer:

- Can a public reader understand what TikiDeco / TIDE is within a few minutes?
- Are current, planned, conceptual, and not-claimable items clearly separated?
- Can technical reviewers verify contracts, manifests, reports, Safe control, and release gates?
- Do the EN/ES/RU texts preserve the same cautious meaning?
- Are the read-only pilot boundaries clear?

## Useful Feedback

- Website clarity, mobile layout, accessibility, and broken read-only links.
- Documentation structure and stale facts.
- Claims that sound too promotional or unsupported.
- Translation issues in EN/ES/RU copy.
- Read-only Sepolia balance-check UX.
- Campaign rules, report templates, blocker gates, and issue triage.
- V2 audit-package readiness questions that do not reveal sensitive details.

## Out Of Scope

- Token offering requests.
- Market-value discussion.
- Mainnet pressure.
- Active hospitality benefit requests.
- Partnership claims.
- Private wallet information, private guest data, or sensitive personal data.
- Public vulnerability details that should go through `SECURITY.md`.

## Maintainer Response Rules

- Acknowledge the feedback directly.
- Link to the source of truth: `PROJECT_FACTS.md`, `CLAIMS_MATRIX.md`, `FEEDBACK_GUIDE.md`, or `COMMUNITY_PREVIEW.md`.
- Classify the item as current, planned, conceptual, not-claimable, or out of scope.
- Do not turn feedback into a promise.
- Create or link an issue when work is needed.
- Ask for evidence when a claim is not supported by repository or on-chain facts.
- Route security-sensitive reports to `SECURITY.md`.

## Weekly Cadence

| Day | Activity |
| --- | --- |
| Monday | Review new issues and apply labels. |
| Wednesday | Update issue status, request missing evidence, and identify release blockers. |
| Friday | Summarize accepted changes, deferred items, out-of-scope topics, and next actions. |

## Feedback Summary Format

Use `docs/COMMUNITY_PREVIEW_REPORT_TEMPLATE.md`.

The summary should include:

- feedback received;
- what is current;
- what is planned;
- what is conceptual;
- what is not claimable;
- changed docs or site copy;
- risks raised;
- next actions.

## Avoid Turning Feedback Into Promises

Use cautious response language:

- "This is planned for review."
- "This requires legal, privacy, operations, or security review."
- "This is not active."
- "This is not supported by current project facts."
- "This can be explored as a documentation improvement."

Avoid:

- timelines framed as commitments;
- statements about market value;
- statements about active hospitality benefits;
- statements about mainnet readiness;
- statements implying independent audit completion.
