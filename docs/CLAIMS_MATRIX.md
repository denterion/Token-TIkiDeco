# TikiDeco Claims Matrix

Verification date: 2026-06-17

Use this matrix together with [`PROJECT_FACTS.md`](PROJECT_FACTS.md). If a claim is not supported by `PROJECT_FACTS.md`, the canonical deployment manifest, repository source, or read-only on-chain data, do not use it in public communications.

## Unified Positioning

English:

> TikiDeco is an open-source Sepolia prototype exploring transparent loyalty and access infrastructure for a future hospitality concept. TIDE currently has no monetary value and is not offered for sale.

Russian:

> TikiDeco - открытый прототип в сети Sepolia, исследующий применение токена для прозрачных программ лояльности и доступа в будущей hospitality-экосистеме. TIDE пока не имеет денежной стоимости и не предлагается к продаже.

## Allowed Claims

These formulations are allowed when used with the same meaning and without implying live mainnet services, financial rights, or completed hospitality operations.

| Formulation | Allowed use | Required context |
| --- | --- | --- |
| Sepolia testnet prototype | Describe the current deployment and public site. | Mention Sepolia/testnet when discussing deployed contracts. |
| Fixed supply in the current deployed contract | Describe the deployed V1 token supply. | Use `100,000,000 TIDE` only for the current deployed contract. |
| Publicly verifiable source code | Link to Sepolia Etherscan source-code pages. | Do not say "audited" or "approved". |
| Experimental loyalty and access infrastructure | Describe the research direction and possible utility scenarios. | Make clear that benefits are not active or guaranteed. |
| No token sale | Describe current availability status. | Do not imply a later sale is approved. |
| No current monetary value | Describe TIDE's current testnet status. | Use with Sepolia/testnet context. |
| Not independently audited | Describe current security/audit status. | Can mention internal review separately, but not as an audit. |

## Claims Requiring Confirmation

Do not publish these claims unless a new verified source is added to `PROJECT_FACTS.md` and legal/compliance review approves the wording when relevant.

| Formulation | Required confirmation before use |
| --- | --- |
| Hotel partnership | Signed/approved public partner confirmation and approved partner messaging. |
| Real-world utility | Live operational rules, legal review, and evidence that the utility is active. |
| Discounts | Published perk terms, operational approval, legal review, and inventory/eligibility rules. |
| Memberships | Membership terms, eligibility criteria, cancellation rules, legal review, and operational process. |
| Property-backed | Legal structure, asset documentation, counsel approval, and explicit risk disclosure. |
| Revenue | Legal review and verified reporting framework; avoid implying holder entitlement. |
| Active users | Analytics or on-chain/off-chain participation data with a defined measurement method. |
| Security audit | Completed independent audit report from a named auditor or firm. |
| Exchange listing | Public confirmation from the exchange or verifiable listing page. |

## Prohibited Claims Until Legal Review

These formulations must not be used in public materials unless qualified counsel approves a compliant structure and the project updates `PROJECT_FACTS.md`.

| Prohibited formulation | Reason |
| --- | --- |
| Guaranteed profit | Implies financial return and is unsupported. |
| Passive income | Implies income rights and is unsupported. |
| Investment opportunity | Implies securities/offering framing and is unsupported. |
| Expected price growth | Implies speculative upside and is unsupported. |
| Risk-free | False or misleading risk framing. |
| Guaranteed listing | Unsupported exchange/listing promise. |
| Ownership share in a hotel | Implies equity or property rights and is unsupported. |
| Guaranteed revenue share | Implies entitlement to revenue. |

## Review Rule

Before publishing new README text, site copy, white paper changes, social posts, or pitch material:

1. Match each claim to [`PROJECT_FACTS.md`](PROJECT_FACTS.md).
2. Label `planned` and `experimental` items clearly.
3. Remove any `unknown` claim.
4. Do not use prohibited claims without legal review.
